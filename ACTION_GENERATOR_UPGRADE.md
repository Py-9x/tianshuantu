# ActionGenerator 升级文档：从被动监测到主动行动指挥

## 问题分析

### 原有系统的缺陷
之前的AI建议存在三类无效内容：

| 问题 | 原建议 | 影响 |
|------|--------|------|
| **监测类无效** | "每2分钟监测一次心率，记录变化" | 系统已自动采集，用户无需手动操作 |
| **泛泛而谈** | "立即检查通信设备电量" | 缺乏具体数值，无法指导行动 |
| **无备用方案** | "补氧，持续5分钟" | 主方案失效时用户陷入困境 |

### 真实高原环境困境
- 用户氧气不足，思维模糊，**无法独立做复杂决策**
- 手指冻僵，**无法频繁操作手机**
- 视线模糊，能见度差，**依靠手机导航不可靠**

**结论**：系统必须替用户做出决策，而不是叫用户去"监测"和"思考"。

---

## 升级方案

### 1. 禁止"监测"类指令（核心改进）

**新要求**：每条建议必须是用户可立即执行的**动作**，禁止：
- ❌ "监测"、"记录"、"检查"（系统已自动做）
- ❌ 被动观察词
- ❌ 泛泛建议（必须量化）

**实施**：在SYSTEM_PROMPT中明确禁止：
```python
【绝对禁止】
- ❌ "监测"或"记录"体征（系统已自动采集，无需用户手动操作）
- ❌ "检查"、"检查一下"这类被动观察词
- ❌ 泛泛建议如"补水"、"休息"（必须给出具体数值如"200ml"、"5分钟"）
```

### 2. 以行动为中心的重构

**原建议**（无效）：
```
"心率79.4bpm，以2.7bpm/min速度上升，预计14.6分钟后接近120bpm警戒值。
每2分钟监测一次心率，记录变化。"
```

**改进后**（有效）：
```json
{
  "title": "降低代谢负荷",
  "detail": "心率从75bpm快速上升到85bpm，以2.7bpm/min速度加快，
           预计14分钟后达到120bpm警戒值。立即停止爬升，原地坐下，
           双脚抬高高于心脏，进行深呼吸（4秒吸/6秒呼）。休息5分钟后
           如心率仍>100bpm，启动下撤预案。",
  "fallback": "若无改善，服用β受体阻滞剂（既往有用药史者），
             并联系救援队。",
  "urgency": "urgent"
}
```

**核心改进点**：
1. ✅ 具体动作（停止、坐下、呼吸模式）
2. ✅ 量化指标（4秒吸/6秒呼、5分钟）
3. ✅ ETA倒计时（14分钟后达警戒值）
4. ✅ 备用方案（若5分钟无改善则…）

### 3. 三层行动分級

| 优先级 | 示例 | urgency |
|--------|------|---------|
| **第1条** | 解决最紧急生命威胁（血氧、保暖） | `urgent` |
| **第2条** | 防止现象恶化（补氧、下撤） | `normal` |
| **第3条** | 通信和撤离准备（发SOS、查地图） | `prepare` |

### 4. 医学依据强化

每条建议都应引用标准指南：
- **HACE**（高原脑水肿）：意识模糊、协调性丧失 → 立即下降1000m+吸氧
- **HAPE**（高原肺水肿）：呼吸困难、粉红泡沫痰 → 吸氧+利尿剂(Lasix 40mg)+下降
- **Hypothermia**（失温）：打寒颤、语言含糊 → 进入背风处、更换干衣、保暖
- **Hypoxia**（缺氧）：SpO2<90%、意识模糊 → 吸氧+深呼吸+紧急下撤

## 代码改动

### A. SYSTEM_PROMPT 重构
**文件**：`views/monitoring.py` 第361行

```python
SYSTEM_PROMPT = """你是野外急救指挥官，基于实时体征趋势和量化指标，
生成3条高度具体、立即可执行的行动指令。

【绝对禁止】
- ❌ "监测"或"记录"体征（系统已自动采集，无需用户手动操作）
- ❌ "检查"、"检查一下"这类被动观察词
- ❌ 泛泛建议如"补水"、"休息"（必须给出具体数值如"200ml"、"5分钟"）

【核心要求】
1. 返回纯JSON数组，绝对不要有任何多余文字或Markdown代码块
2. 每条必须包含4个字段：title（≤8字）、detail（含具体数值和倒计时）、
   fallback（备用方案）、urgency（urgent/normal/prepare）
3. 第1条解决最紧急的生命威胁，第2条防止恶化，第3条通信/撤离准备
4. detail中必须引用输入中的具体数值，并给出用户应立即执行的动作
5. fallback中提供主方案无法执行时的替代措施
6. urgency必须是'urgent'或'normal'或'prepare'

【医学参考】
...(HACE、HAPE、失温、缺氧的标准定义及应对)...
"""
```

## 效果对比

### 血氧快速下降场景

**改进前** (问题多):
```
"血氧96.4%，以-1.5%/min速度下降。建议每2分钟复测血氧，
记录变化趋势。如出现头晕、嘴唇发紫症状立即停止运动。"
```
问题：
- ❌ 叫用户"记录"（事已至此还手工记录？）
- ❌ 缺乏倒计时（多久会危险？）
- ❌ 无具体行动指令
- ❌ 无备用方案

**改进后** (国赛级):
```json
{
  "title": "立即停止上升",
  "detail": "血氧96.4%，以-1.5%/min速度下降，LSTM预测4分钟后跌破90%。
           立即原地停止，盘腿坐下，双脚抬高心脏，深呼吸4秒吸/6秒呼，
           同时检查背包中的便携式供氧瓶。",
  "fallback": "若5分钟内血氧未回升至92%以上，或出现头晕、嘴唇发紫，
             立即启动下降至少300米、同时触发SOS对讲机（频道VHF.A）。",
  "urgency": "urgent"
}
```
优点：
- ✅ 明确倒计时（4分钟）
- ✅ 具体动作（盘腿、抬腿、呼吸节奏）
- ✅ 备用触发条件（5分钟未改善则下撤）
- ✅ 医学依据清晰

## 后续可优化方向

当用户配有心率手环时，提升数据实时性：
- 实时心率变化速率更精确

- 自动发送位置到基地

### 2. 离线语音指导（PWA模式）
在网络差的地方，应用启动时缓存的语音播放：
```python
def _play_emergency_instruction(emergency_type: str):
    """
    高危情况下自动播放离线语音指导
    用户无需看手机，直接听指令执行
    """
    if emergency_type == "HAPE":
        st.audio("assets/hape_instruction.mp3", autoplay=True)
        # "停止运动，坐下，深呼吸，准备下撤..."
```

### 3. 纸质应急卡片生成
用户完成profile后生成防水PDF，包含关键指令：
```
【 高原应急卡片 】

🔴 HACE症状（头晕/脑胀）
→ 停止向上
→ 坐下深呼吸
→ 服用地塞米松4mg
→ 发送SOS信号

🟡 HAPE症状（呼吸困难）
→ 立即停止运动
→ 半卧位休息
```

- 队员A血氧下降速率异常

- 队员C已3小时无位置更新
